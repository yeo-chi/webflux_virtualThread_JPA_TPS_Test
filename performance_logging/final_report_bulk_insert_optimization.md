# 최종 성능 최적화 결과 보고서 (v2.0)
- **날짜**: 2026-02-15
- **주제**: 세마포어(Semaphore) 및 트랜잭션 수명 주기 최적화 후 최종 검증

## 1. 추가 최적화 내역
- **리소스 격리 (Semaphore)**: 커넥션 풀(100개) 고갈 방지를 위해 메인 비즈니스(60), 어딧 로직(30)으로 동시 실행 수 제한.
- **트랜잭션 제어**: `@Transactional` 대신 `TransactionTemplate`을 사용하여 세마포어 획득 이후에만 DB 커넥션을 점유하도록 구조 변경.
- **대기열 관리**: 가상 스레드(Virtual Thread)를 활용하여 세마포어 대기 시 컨텍스트 스위칭 비용 최소화.

## 2. 최종 성능 테스트 결과 (700 TPS 부하)

| 지표 | 세마포어 적용 전 | 세마포어 + 트랜잭션 최적화 후 |
| :--- | :--- | :--- |
| **성공률 (Success Rate)** | ~81% (랜덤한 500 에러 발생) | **100.00% (장애 완전 해소)** |
| **평균 지연 시간** | ~1,138ms | **~111ms (약 10배 개선)** |
| **P95 지연 시간** | ~5,292ms | **~220ms (매우 안정적)** |
| **실제 처리량 (RPS)** | ~130 RPS (지연 누적) | **~697 RPS (목표치 달성)** |

## 3. 700 TPS를 버틸 수 있는 이유 (수학적 근거)
1. **작업 시간 단축**: 벌크 인서트로 인해 1건의 요청이 커넥션을 점유하는 시간이 약 100ms 내외로 최적화됨.
2. **이론적 용량**: 풀 사이즈 100 기준, 전체 작업이 100ms일 때 초당 최대 1,000건 처리 가능 ($100 / 0.1s$).
3. **세마포어의 역할**: 초당 700건이 몰려도 커넥션 풀(100개)이 감당 가능한 범위 내로 유입을 조정(Throttling)함. 남은 요청은 에러가 나는 대신 가상 스레드에서 아주 짧게 대기한 후 순차적으로 처리됨.

## 4. 결론 및 향후 과제
- **결론**: 현재 코드 수준에서 **700 TPS는 충분히 안정적으로 수용 가능**함.
- **과제**: 지속적인 700 TPS 부하 시 DB 서버의 CPU/디스크 I/O 사용률을 모니터링하여 인프라 증설 시점을 판단해야 함 (초당 35,000건의 로그 인서트 데이터 누적 대비).
